34 /**
35  * @file
36  * Event Machine timer add-on
37  * @defgroup em_timer Event timer add-on
38  * Event Machine timer add-on
39  * @{
40  *
41  * The timer API can be used to request an event to be sent to a specified
42  * queue at a specified time once (one-shot) or at regular intervals (periodic).
43  * A timer needs to be created first - it represents a collection of timeouts
44  * with certain attributes (e.g. timeout resolution and maximum period).
45  * A timer can be mapped to a HW resource on an SoC, thus the number of timers,
46  * capabilities and time bases are system specific. Typically only one or a few
47  * timers are supported.
48  * The application can specify required capabilities when a timer is created.
49  * The creation will fail if the implementation cannot fulfill the required
50  * values. Timers are typically created once at system startup.
51  *
52  * A timer is a shared resource with proper synchronization for concurrent
53  * multi-thread use. It is possible to exclude all multi-thread protections if a
54  * timer is used exclusively by a single thread (for potential performance
55  * gains). This is done by setting EM_TIMER_FLAG_PRIVATE when creating a timer.
56  * Setting this flag means that the application must ensure that only a single
57  * thread is using the timer (this also includes the receiver of periodic
58  * timeouts due to the ack-functionality). This private-mode is not necessarily
59  * implemented on all systems, in which case the flag is ignored as it will not
60  * cause any functional difference.
61  *
62  * Timeouts (tmo) can be created once a timer exists. Creating a timeout
63  * allocates the resources needed to serve the timeout, but does not arm it -
64  * this makes it possible to pre-create timeout(s) and set the expiry
65  * later at runtime. This can improve performance but also minimizes the
66  * possibility that the runtime call setting the expiry would fail, as resources
67  * have already been reserved beforehand.
68  *
69  * A pending timeout can be cancelled. Note that there is no way to cancel an
70  * expired timeout for which the event has already been sent but not yet
71  * received by the application - canceling, in this case, will return an error
72  * to enable the the application to detect the situation. For a periodic timer,
73  * a cancel will stop further timeouts, but may not be able to prevent the
74  * latest event from being received.
75  *
76  * An active timeout cannot be altered without canceling it first.
77  *
78  * Timeouts needs to be deleted after use. Deletion frees the resources reserved
79  * during creation.
80  *
81  * The unit of time (seconds, nanoseconds, cycles of 1.66GHz clock, frame
82  * numbers etc.) is defined per timer/clock source and is system dependent.
83  * The timeout value is an abstract "tick" defined per timer.
84  * A clock source can be specified when creating a timer. It defines the time
85  * base of the timer for systems with multiple sources implemented.
86  * EM_TIMER_CLKSRC_DEFAULT is a portable value that implements a basic
87  * monotonic non-wrapping time, typically based on the CPU clock.
88  * It is assumed that the tick count is monotonic and will not wrap around,
89  * unless a specific timer uses something exotic like a cyclic short frame
90  * number counter.
91  * The tick frequency of a timer can be inquired at runtime.
92  *
93  * A periodic timer requires the application to acknowledge each received
94  * timeout event after it has been processed. The acknowledgment activates the
95  * next timeout and compensates for the processing delay to keep the original
96  * interval. This creates a flow control mechanism and also protects the event
97  * handling from races if the same event is reused every time - the next
98  * timeout will not be sent before the previous has been acknowledged.
99  * The event to be received for each periodic timeout can also be different,
100  * because the next event is always given by the application through the
101  * acknowledge operation.
102  * The target queue cannot be modified after the timeout has been created.
103  *
104  * If the acknowledgment of a periodic timeout is done too late (after the next
105  * period has already passed), the default action is to skip the missed timeout
106  * slot(s) and arm for the next valid slot. If the application never wants to
107  * skip a missed timeout it can set the flag EM_TMO_FLAG_NOSKIP when creating a
108  * timeout, which causes each acknowledgment to schedule an immediate timeout
109  * event until all the missed time slots have been served. This keeps the number
110  * of timeouts as expected but may cause an event storm if a long processing
111  * delay has occurred.
112  *
113  * The timeout handle is needed when acknowledging a periodic timeout event.
114  * Because any event can be used for the timeout, the application must itself
115  * provide a way to derive the timeout handle from the received timeout event
116  * (a typical way is to include the tmo handle within the timeout event).
117  * The application also needs to have a mechanism to detect which event is a
118  * periodic timeout to be able to acknowledge.
119  *
120  * Example usage
121  * @code
122  *
123  * // This would typically be done at application init.
124  * // Accept all defaults but require at least 100us resolution
125  * // and give a name.
126  * em_timer_attr_t attr;
127  * memset(&attr, 0, sizeof(em_timer_attr_t));
128  * attr.resolution = 100000;
129  * strncpy(attr.name, "100usTimer", EM_TIMER_NAME_LEN);
130  * em_timer_t tmr = em_timer_create(&attr);
131  * if(tmr == EM_TIMER_UNDEF) {
132  * // out of resources or too tight requirements
133  * // handle error here or via error handler
134  * }
135  *
136  * // At runtime - create a timeout resource.
137  * // Can be done in advance to save time if the target queue is known.
138  * em_tmo_t tmo = em_tmo_create(tmr, EM_TIMER_FLAG_ONESHOT, target_queue);
139  * if(tmo == EM_TMO_UNDEF) {
140  * // no such timer or out of resources
141  * // handle error here or via error handler
142  * }
143  *
144  * // Get the timer frequency
145  * uint64_t hz = em_timer_get_freq(tmr);
146  *
147  * // Activate a 10ms timeout from now.
148  * // Very unlikely to fail with valid arguments.
149  * if (em_tmo_set_rel(tmo, hz / 100, my_tmo_event) != EM_OK) {
150  * // handle error here or via error handler
151  * }
152  *
153  * @endcode
154  *
155  * At the moment the timer is an 'add-on', i.e. not part of the base API exposed
156  * by event_machine.h
157  *
158  */

225 /**
226  * Structure used to create a timer (or inquire its capabilities).
227  *
228  * Setting a field to 0 means that the application accepts the default value.
229  * Setting a non-zero value indicates a requirement. E.g. setting 'resolution'
230  * to 1000(ns) requires at least 1us resolution. The timer creation will fail if
231  * the implementation cannot support such high a resolution .
232  * The implementation is free to provide better than requested, but not worse.
233  */
234 typedef struct em_timer_attr_t {
235  /** Timeout min resolution, ns. 0 for default */
236  uint64_t resolution;
237  /** Maximum timeout period, ns. 0 for default */
238  uint64_t max_tmo;
239  /** Min simultaneous timeouts. 0 for default */
240  uint32_t num_tmo;
241  /** Extra flags. A set flag is a requirement */
242  em_timer_flag_t flags;
243  /** Clock source (system specific). 0 for default */
244  em_timer_clksrc_t clk_src;
245  /** Optional name for this timer */
246  char name[EM_TIMER_NAME_LEN];
247 } em_timer_attr_t;

322 /**
323  * Activate a timeout (absolute time) with a user-provided timeout event
324  *
325  * Sets the timeout to expire at a specific absolute time. The timeout event
326  * will be sent to the queue given to em_tmo_create() when the timeout expires.
327  *
328  * It is not possible to send timeouts with an event group (but the application
329  * can assign the event group when receiving the timeout event,
330  * see em_event_group_assign()).
331  *
332  * The timeout event should not be accessed after it has been given to the
333  * timer.
334  *
335  * Even if not guaranteed, the implementation should make sure that this call
336  * can fail only in highly exceptional situations (em_tmo_create() should pre-
337  * allocate needed resources).
338  *
339  * Setting 'ticks' to an already passed time value is considered an error.
340  *
341  * An active timeout can not be modified. The timeout needs to be canceled and
342  * then set again (tmo handle can be re-used) with new arguments.
343  *
344  * An inactive timeout can be re-used by calling em_tmo_set_abs/rel() again
345  * after the previous timeout has expired.
346  *
347  * This function cannot be used to activate a periodic timer, instead use
348  * em_tmo_set_rel().
349  *
350  * @param tmo Timeout handle
351  * @param ticks_abs Expiration time in absolute timer specific ticks
352  * @param tmo_ev Timeout event
353  *
354  * @return EM_OK on success
355  */
356 em_status_t em_tmo_set_abs(em_tmo_t tmo, em_timer_tick_t ticks_abs,
357  em_event_t tmo_ev);

401 /**
402  * Acknowledge a periodic timeout.
403  *
404  * All received periodic timeout events must be acknowledged with em_tmo_ack().
405  * No further timeout event(s) will be sent before the user has acknowledged
406  * the previous ones.
407  *
408  * Timeout acknowledgment is usually done at the end of the EO-receive function
409  * to prevent race conditions (e.g. if the same event is re-used for the next
410  * timeout period also). The implementation will adjust for the processing delay
411  * so that the time slot will not slip/drift over time.
412  *
413  * If em_tmo_ack() is called "too late", e.g. the next period(s) are already
414  * passed, the implementation will by default skip all the missed time slots and
415  * arm for the next future one. The application can, if skipping is undesired,
416  * alter this behaviour with the flag EM_TMO_FLAG_NOSKIP when creating a
417  * timeout - here no past timeout is skipped and each acknowledgment will
418  * immediately trigger the next timeout until the current time has been reached.
419  * Note that using EM_TMO_FLAG_NOSKIP may result in an event storm if a large
420  * number of timeouts have been unacknowledged for a longer time.
421  *
422  * The application may re-use the same timeout event that was received or
423  * provide a new one for the next timeout.
424  * An error will be returned if the corresponding timeout has been cancelled.
425  *
426  * The given event should not be touched after calling this until it has been
427  * received again.
428  *
429  * The given event must be handled by the application if an error is returned.
430  * The periodic timeout will stop if em_tmo_ack() returns an error.
431  *
432  * em_tmo_ack() can only be used with periodic timeouts and will fail for
433  * one-shot timeouts.
434  *
435  * @param tmo Timeout handle
436  * @param next_tmo_ev Next timeout event handle (can be the current one)
437  *
438  * @return EM_OK on success
439  */
440 em_status_t em_tmo_ack(em_tmo_t tmo, em_event_t next_tmo_ev);
